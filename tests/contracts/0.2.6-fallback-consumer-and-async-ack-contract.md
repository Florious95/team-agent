# 0.2.6 Fallback Consumer And Async Ack Contract

This contract pins two user-approved 0.2.6 fixes:

- Bug 52: durable fallback messages must have a consumer and unambiguous
  success semantics.
- Bug 55: worker-to-worker MCP sends are asynchronous and must return an
  explicit pending acknowledgement instead of legacy mixed-status fields.

## Bug 52: Leader Inbox Fallback Consumer

Old behavior:

- Worker-to-leader fallback wrote `.team/runtime/leader-inbox.log` but returned
  `ok: false, status: "fallback", durably_stored: true`.
- Worker-to-worker fallback used a different envelope for the same physical
  fact: durable storage succeeded.
- The leader inbox log had no automatic consumer, so a leader could repeatedly
  run normal `team-agent` commands and never see the fallback entry.

New behavior:

- Durable fallback storage is success. If a fallback entry is written to
  `.team/runtime/leader-inbox.log`, the response has `ok: true`.
- `ok: false` is reserved for cases where no durable storage happened.
- Worker-to-leader and worker-to-worker fallback responses use the same
  success envelope shape:
  `ok: true`, `status: "queued"` or `status: "fallback_log"`,
  `message_id`, and `fallback_path`.
- Every `team-agent <verb>` CLI invocation checks
  `.team/runtime/leader-inbox.log` before printing the command's normal output.
  If the file has bytes beyond `.team/runtime/leader-inbox.cursor`, the CLI
  prepends a bounded summary to the top of stdout or stderr.
- After a summary is printed, `.team/runtime/leader-inbox.cursor` advances to
  the inbox EOF. The next CLI invocation prints only newer entries.

Summary schema:

- `count`: number of new fallback entries consumed.
- `items`: recent entry titles or first 80 content characters.
- `hint`: contains `team-agent inbox <id>` so the leader can inspect full
  content.
- `truncated`: true when the summary had to drop or shorten content to fit the
  budget.

Mechanical acceptance:

- Write two entries to `.team/runtime/leader-inbox.log`, run
  `team-agent status --workspace <workspace>`, and assert the first output
  block is a leader-inbox summary containing the new-entry count and an inbox
  command hint.
- Run the same command twice. The first run consumes old entries and advances
  `.team/runtime/leader-inbox.cursor` to EOF; the second run does not repeat
  the old entries.
- Force worker-to-leader delivery down the fallback path by making the attached
  leader pane invalid. The public send result must be `ok: true` with
  `status in {"queued", "fallback_log"}` and a fallback log file path.
- With many long fallback entries, the prepended summary must remain at or
  below 500 characters, indicate truncation, and point to `team-agent inbox`.

## Bug 55: MCP Async Ack

Old behavior:

- Worker-to-worker `team_orchestrator.send_message` returned a snapshot such
  as `ok: true`, `status: "queued"`, `submitted: false`, `visible: false`,
  `queued: true`, and `durably_stored: true`.
- Sender models interpreted `submitted: false` as a failure and could resend,
  creating duplicate messages.

New behavior:

- Worker-to-worker MCP `send_message` returns immediately after accepting the
  message for asynchronous delivery. It does not wait for tmux paste
  verification.
- The response fields are intentionally small and non-overlapping:
  `status: "accepted"`, `delivery_pending: true`,
  `poll_via: "team-agent inbox <id>"`.
- `message_id` may be included so the sender can poll the exact message.
- Historical fields are forbidden in this response:
  `ok`, `submitted`, `visible`, `queued`, and `durably_stored`.
- Old workers that depended on `submitted` or `ok` must upgrade. This is a
  breaking protocol cleanup because the old fields caused ambiguous retries.

Mechanical acceptance:

- Call MCP `TeamOrchestratorTools.send_message` from one worker to another and
  assert the returned keys are exactly
  `{status, delivery_pending, poll_via}` or those plus `message_id`.
- Assert `status == "accepted"` and `delivery_pending is True`.
- Patch the synchronous delivery path to sleep or fail if called, then assert
  the MCP call returns within 100 ms and does not wait for paste verification.
- Assert `poll_via` is exactly in the format `team-agent inbox <id>` and the
  `<id>` portion is non-empty and parseable.
