# 0.2.6 Positive Source Rebind Contract

This contract pins the blocking constraints from
`.team/artifacts/0.2.6-main-slice-cr-verdict.md`. The governing rule is
Constitution `MUST-11` plus `MUST-NOT-12`: caller, active team, and sender
scope facts must come from positive authoritative sources, never reverse
enumeration plus heuristic ranking.

## Family A: Shared Caller-Pane Binding

Positive source: `$TMUX_PANE`.

- C1: `bind_owner_from_caller_pane(workspace, team_id, override_uuid=None)` is
  the only owner-binding function. `takeover`, `claim-leader`, and
  `quick-start` all call it. Acceptance: source/AST test finds one shared call
  path and no independently derived owner identity in those entry points.
- C2: the bind function reads `$TMUX_PANE` and checks only that pane with
  `tmux display-message -p -t $TMUX_PANE '#{pane_current_command}'`.
  Acceptance: mock tmux records no more than two calls, every `-t` target equals
  `$TMUX_PANE`, and no `list-panes`, `list-windows`, or `list-clients` occurs.
- C3: no `$TMUX_PANE` means refusal with a user hint to run from the leader
  pane. Acceptance: unset env, call bind, assert refused result and event.
- C4: caller pane current command that is not `claude`, `claude.exe`, `codex`,
  or a registered leader host is refused at bind time. Acceptance: real tmux
  fixture uses a `broot` pane and asserts no state mutation plus
  `owner.bind_refused`.
- C5: successful bind force-writes every `team_owner` identity field from the
  caller-derived identity; it does not merge, migrate, or setdefault old values.
  Acceptance: old owner U1 plus caller U2, then `takeover --confirm`, leaves all
  owner fields equal to U2 and is idempotent on repeat.
- C24: code paths in `state.py`, `leader_panes.py`, `messaging/*`,
  `mcp_server*`, and `cli/parser.py` do not contain reverse enumeration,
  heuristic sort, and take-first patterns for binding/candidate/scope decisions.
  Acceptance: lint test greps forbidden idioms.
- C25: contract-level tmux mock enforces C2 mechanically.

Audit event: every successful bind emits
`owner.bound_from_caller_pane {caller_pane_id, caller_current_command,
derived_uuid_prefix, team_id, old_uuid_prefix}`.

Refusal event: every C3/C4 refusal emits
`owner.bind_refused {reason, caller_pane_id, caller_current_command, hint}`.

## Family B: Active Team And Candidate Source

Positive source: `state.active_team_key` plus alive entries in `state.teams`.

- C6: state schema has `active_team_key: str | None`; legacy top-level
  `session_name` may seed it once, then explicit mutations own it. Acceptance:
  load legacy state and assert persisted `active_team_key`.
- C7: `team_state_candidates(state)` reads only `state.teams` filtered by
  `status == "alive"` and never uses `setdefault(team_state_key(...),
  compact_team_state(...))`. Acceptance: state with A/B alive and top-level C
  shutdown returns A/B only; grep test rejects the old idiom.
- C8: top-level `session_name` and `team_dir` are a derived view of
  `teams[active_team_key]`, not an independently writable source. Acceptance:
  top-level C plus active B selects B.
- C9: `send <agent>` without `--team` scopes to `active_team_key`; if no active
  key it refuses with quick-start hint; if multiple alive teams and no active
  key it is ambiguous but lists only alive teams. Acceptance: resolve/send
  matrix.
- C10: shutdown synchronously and atomically deletes `state.teams[team_key]`,
  clears `active_team_key` if it pointed there, and archives
  `.team/runtime/teams/<session>/` to `.archived-<session>-<ts>/`.
- C11: `--keep-logs` means logs remain readable in the archive, but state
  registration and directory archival still happen.
- C12: shutdown cleanup is not async; immediately after shutdown returns, C10
  is complete or the pre-state remains intact.

Audit events: successful shutdown emits
`team.shutdown_completed {team_key, archive_path, teams_remaining,
new_active_team_key}`; blocked shutdown emits
`team.shutdown_blocked {reason}`.

## Family C: MCP Sender Scope

Positive source: worker spawn-time `TEAM_AGENT_OWNER_TEAM_ID`.

- C13: worker launch env always includes `TEAM_AGENT_OWNER_TEAM_ID`; an existing
  value is verified against the owning team. Acceptance: spawn plan/env test.
- C14: MCP `send_message` server reads sender team id from MCP context/env and
  passes it as implicit scope. Workers pass only `to=<name>`, not team names.
  Acceptance: same-team target resolves without `team_target_ambiguous`.
- C15: `to="*"` defaults to team scope. Cross-team broadcast requires explicit
  `scope="workspace"`. Acceptance: default star is not workspace-wide.
- C16: server-side visible peers/state/inbox are filtered to sender team alive
  agents only. Other teams and dead agents are not returned or named in hints.
- C17: candidate resolution source is only sender injected team id; MCP server
  does not infer scope by scanning state, inbox, events, or sorting candidates.

Audit event: every MCP resolution emits
`mcp.scope_resolved {sender_team_id, requested_to, resolved_agent, scope}`.

Refusal events: C9/C16/C17 refusals emit a closed reason enum and a next-step
hint. Cross-team names must not leak in refusal text unless `scope=workspace`
was explicitly requested.

## Audit Schema Summary

- `owner.bound_from_caller_pane`: `caller_pane_id`, `caller_current_command`,
  `derived_uuid_prefix`, `team_id`, `old_uuid_prefix`.
- `owner.bind_refused`: `reason`, `caller_pane_id`, `caller_current_command`,
  `hint`.
- `team.shutdown_completed`: `team_key`, `archive_path`, `teams_remaining`,
  `new_active_team_key`.
- `team.shutdown_blocked`: `reason`, `team_key`, `hint`.
- `mcp.scope_resolved`: `sender_team_id`, `requested_to`, `resolved_agent`,
  `scope`.
- All refusal-path events: `reason` uses a closed enum and `hint` tells the user
  the next action. Silent fallback is a contract failure.
