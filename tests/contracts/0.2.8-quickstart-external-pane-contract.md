# 0.2.8 Pane-As-Identity Migration Contract

## Scope

This contract covers the complete 0.2.8 pane-as-identity migration for bugs
064-068. Multiple leader identity paths can locate a live tmux pane, but reject
an externally launched Claude Code pane because command names, deterministic
UUIDs, or injected `TEAM_AGENT_LEADER_*` env are treated as authorization
identity. Pane id is the leader identity. `owner_epoch` is the lease version.
Command names, provider, machine fingerprint, deterministic UUID, and injected
env are metadata and ranking hints only.

## Contract

1. `_pane_is_usable_leader` must not reject by command name. If the pane cwd
   matches the workspace, commands such as `2.1.154`, empty string, `node`, or
   unknown binaries are usable leader panes.

2. `_resolve_leader_pane(None, provider, workspace, require_current=True)` must
   adopt the current tmux client pane when its cwd equals the workspace, with
   discovery mode `current_client`, regardless of `pane_current_command`.

3. Workspace isolation remains binding. A pane whose cwd does not equal the
   workspace must still be rejected, even if command-name gating is removed.

4. The quick-start / launch error path must not recommend a nonexistent
   `--pane` option unless quick-start actually supports that option. The
   failure text should describe the concrete mismatch, such as cwd mismatch or
   missing current tmux pane.

5. Worker-to-leader delivery and receiver claim paths must not reject a live
   leader pane solely because `pane_current_command` is `2.1.154`. In
   particular, the receiver validation path must not return
   `leader_pane_wrong_command`, first-time leader binding must not return
   `leader_pane_wrong_command`, and receiver claim must not return
   `wrong_command` for that command.

6. `_leader_command_looks_usable(command, provider)` is no longer an
   allowlist. It returns `True` for any non-empty command string, including
   `2.1.154`. Empty command strings remain not usable so the runtime does not
   bind a pane whose process command could not be observed.

7. Leader lease helpers used by attach / receiver binding / re-resolve must
   treat `2.1.154` as live leader evidence when cwd/identity checks pass. Exact
   command matching may still be used for ranking preference, but never as a
   hard rejection gate.

8. Receiver binding authorizes by pane equality when the target pane is the
   recorded owner pane. If `receiver.pane_id == team_owner.pane_id`, the runtime
   must bind/validate the receiver even when the pane has no injected
   `TEAM_AGENT_LEADER_SESSION_UUID`; it must not return `leader_uuid_missing`.

9. Receiver claim authorizes an uninstrumented owner pane by pane equality. A
   candidate pane equal to `team_owner.pane_id` must not be refused as
   `uuid_mismatch` merely because it lacks injected uuid env.

10. Pane equality does not remove workspace isolation. A different live pane
   must not be allowed to claim/bind over a live owner pane just because uuid is
   absent.

11. Restart must preserve team identity fields. After a successful restart,
   top-level `spec_path`, `session_name`, and `team_dir` remain populated or are
   derivable from the selected team entry.

12. Send must recover the team spec when `state.spec_path` is absent but
   `team_dir/team.spec.yaml` exists. Missing top-level `spec_path` must not make
   valid workers look outside the team and fail as `target_not_in_team`.

13. Receiver validation has one source of truth: a pane-only bound-leader check
    equivalent to `validate_bound_leader_pane`. All receiver validation paths
    authorize by `leader_receiver.pane_id == team_owner.pane_id` plus live pane
    evidence. They must not emit `leader_uuid_missing`,
    `leader_uuid_mismatch`, or `uuid_mismatch`.

14. Claim, readopt, rediscovery, attach, restart autobind, and worker-to-leader
    delivery must not compare UUIDs for authorization. A live different pane may
    still be refused unless explicitly claimed with the normal workspace and
    confirmation safety gates.

15. Every lease mutation writes `team_owner` and `leader_receiver` together
    through the single dual-state lease writer. After quick-start, takeover,
    attach-leader, claim-leader, and restart autobind, workspace state,
    `state.teams[team]`, and team-scoped state agree on `pane_id` and
    `owner_epoch`.

16. Restart preserves selected-team identity. `spec_path`, `session_name`,
    `team_dir`, and `active_team_key` survive restart and stale-receiver cleanup
    or are derivable from the selected team entry before any send path loads the
    spec.

17. Result notification dedupe is keyed by `(result_id, owner_team_id,
    owner_epoch)`. `leader_session_uuid` may remain nullable legacy/audit
    metadata during migration, but it is not part of the primary or unique key.
    Duplicate result notifications in the same team and epoch inject once; after
    an explicit pane claim advances `owner_epoch`, the same result may notify the
    newly claimed leader pane once.

18. The team.db schema migration must rebuild legacy
    `(result_id, leader_session_uuid)` leader notification logs into the
    team/epoch layout without dropping rows. Existing rows receive a deterministic
    compatible `owner_epoch` value and remain inspectable.

19. Worker-to-leader delivery for an external official Claude pane with no
    injected UUID env must call the live tmux injection path for that pane,
    return `ok=true`, `status=submitted`, and `channel=direct_tmux`, and must not
    create a leader-inbox fallback entry.

20. Restart must persist the selected team key as the team selection truth
    source. After `restart --team current` or an equivalent selected restart,
    `state.teams["current"]` exists and carries the same `spec_path`,
    `session_name`, `team_dir`, `team_owner`, and `leader_receiver` as the
    top-level selected-team view. Explicit `select_runtime_state(workspace,
    "current")` must resolve this entry and must not fail with
    `team 'current' not found`.

21. Post-restart result notifications keep their original `owner_team_id`.
    A resumed worker reporting a result for owner team `current` causes
    coordinator/internal delivery to resolve `team="current"` successfully and
    proceed to the leader notification path. The fix must preserve the
    `owner_team_id=current` contract; it must not paper over the bug by dropping
    team scoping or rewriting the scheduled event to a different key.

## Acceptance

Executable acceptance lives in
`tests/test_028_quickstart_external_pane_acceptance.py`. It uses the real
public runtime seams and tmux-pane capture format from `_legacy_pane_discovery`
instead of invented helpers. Real-machine failure evidence for the external
Claude no-UUID pane lives at
`tests/fixtures/pane_as_identity/external-claude-no-uuid-e2e-evidence.txt`.

## End-To-End Real-Machine Acceptance

This acceptance is required before 0.2.8 ship and is not substitutable by unit
tests, fake tmux, mocked `_tmux_inject_text`, or a pane with pre-injected
`TEAM_AGENT_LEADER_SESSION_UUID`.

1. Start official Claude Code externally in a tmux pane inside the workspace.
   The pane command may be the version string `2.1.154` or equivalent, and the
   pane must have no `TEAM_AGENT_LEADER_SESSION_UUID` in the discovered leader
   env.
2. Run quick-start, attach-leader, or restart from that visible leader pane with
   `$TMUX_PANE=<pane>`.
3. From a worker MCP session, call `send_message(to="leader", content="E2E pane
   identity hello <unique-token>")`.
4. The unique token appears in the external Claude leader pane capture.
5. The send result is `ok=true`, `status=submitted`, `channel=direct_tmux`, and
   `leader_receiver.pane_id=<pane>`.
6. `.team/runtime/leader-inbox.log` does not receive a fallback entry containing
   the token.
7. Events include `leader_receiver.deliver_attempt` and
   `leader_receiver.submitted` for the pane and do not include
   `leader_uuid_missing`, `leader_receiver.delivery_failed`, or
   `leader_not_attached` for the message.
8. After `team-agent restart --team <team>`, repeating the worker send reaches
   the same live pane or a newly claimed current pane, with workspace state,
   `state.teams[team]`, and team-scoped state still agreeing on pane and epoch.
