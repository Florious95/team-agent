## New product bug — F8

### F8: Launch.rs seed-state ordering skips spec compile
- **Symptom**: quick-start's Batch 8 seed-state block (writes `active_team_key + transport.kind` to state.json BEFORE calling `start_coordinator`) creates a state.json that looks like an existing runtime to the downstream launch code. The downstream "compile spec + spawn workers" path is skipped, so `send` gets `target_not_in_team`.
- **Root cause**: Batch 8 fix2 seeded state to give the coordinator daemon its team_key at boot, but the downstream launch code interprets any `state.active_team_key` as "already launched, use restart".
- **Fix options**:
  - **A**: Move seed-state AFTER spec compile but BEFORE `start_coordinator`. Requires re-plumbing the launch flow (invasive).
  - **B**: Pass `--team` explicitly to the coord spawn command line so the daemon doesn't need to derive from state. Small localized change to `start_coordinator` + `parse_args`.
  - **C**: Delay coord start until after spec compile completes; workers spawn against the current-process ConPtyBackend (from spawn+detach); coord starts + reconnects when it boots up next. Similar to Batch 6 direct-handoff pattern but without the F4 workaround.
- **Recommendation**: Option B (pass team via CLI). Cleanest; no re-plumbing.
