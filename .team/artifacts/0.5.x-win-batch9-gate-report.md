## CHECK 5 (peek) partial — architectural note

`peek` runs in the CLI process (not through the coord daemon's MCP). It opens its own `ConPtyBackend` via `resolve_transport`, which tries to connect a fresh `NamedPipeClient` to the shim's pipe. But the shim was owned + spawned by the daemon; when the daemon connects its own client, the shim's single-connection MVP loop is occupied. Subsequent CLI-side connects race and fail.

Distinct from F7/F8:
- F7 was "shim didn't survive quick-start exit" — fixed by DETACHED_PROCESS + coord ownership.
- F8 was "coord daemon booted with wrong team_key" — fixed in this batch.
- **CHECK 5 partial = "CLI + daemon can't share the shim's single-connection pipe"** — needs the shim's Phase 3 multi-connection multiplex OR a CLI → daemon MCP tunnel (peek/capture routes through daemon's already-wired backend).

**Fix estimate**: ~200 LoC in `windows_shim.rs` (add per-request accept + inner-loop reuse) OR add MCP route for `peek`/`capture` in `messaging/`. Batch 10 target.
