# A. Gap 26 - Deterministic SHA256 UUID + Sticky Binding + Claim-Leader Broadcast

## A1. What To Test

In plain words:

When the user runs a team, the leader identity must be stable across terminal pane churn and must not be stolen by a branch window.

The user should experience this as:

- "I switch Ghostty windows; worker results still find my leader."
- "I open a branch of the same conversation; the branch cannot mutate the existing team."
- "If two windows both qualify after the old leader disappears, both receive one clear recovery prompt."
- "When I run `claim-leader --confirm` in one candidate, that window becomes the leader and the others are refused."

Runtime behavior to guarantee:

- `leader_session_uuid = sha256(machine_fingerprint || workspace_abspath || os_user || team_id)[:32]`.
- `pane_id` is a transport hint, not identity.
- A valid bound pane suppresses scanning.
- Side panes are refused while a valid bound pane exists.
- Invalid bound pane causes rediscovery.
- One candidate auto-binds.
- Multiple candidates broadcast one deduped incident.
- Workers do not inherit leader UUID env.
- CLI exposes recovery and diagnostics.

# B. Gap 32 - Same Result Envelope Dedupe

## B1. What To Test

In plain words:

When one worker reports one result, the leader sees that exact result once.

The user should not see three copies because:

- watcher retry created a new message id;
- rebind retried after stale pane recovery;
- multiple watcher rows existed for one result;
- coordinator tick ran again;
- report_result was submitted twice with the same result id.

Runtime behavior to guarantee:

- Dedupe occurs at notification-message level, not only watcher-row level.
- Successful delivery records a stable `notified_message_id` or equivalent.
- Duplicate paths emit an audit event and skip injection.
- Rebind retry carries the same `result_id` and does not become a second visible envelope.

# C. Gap 16 - `team-agent remove-agent` CLI With Atomic Five-Location Cleanup

## C1. What To Test

In plain words:

When the user removes a worker, it is gone everywhere. If removal is refused or fails halfway, the team is not half-edited.

The five storage points are:

- workspace-level `state.json`;
- team-level `state.json`;
- `team.spec.yaml`;
- `.team/dynamic-role-files/<id>.md`;
- `agent_health` row.

Runtime behavior to guarantee:

- Dynamic remove clears all five locations.
- Restart/status does not resurrect removed agents.
- Spec-native removal refuses unless `--from-spec --confirm`.
- Running worker removal refuses unless `--force`.
- `--force` stops first.
- If later cleanup fails, forced stop is rolled back.
- Rollback failure is reported with resource evidence.
