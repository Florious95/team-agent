## New product bug surfaced (Batch 7.5 candidate)

### F7: Daemon-shim coexistence gap
- **Symptom**: `send` returns `coordinator_unavailable` even after `team-agent coordinator --workspace ...` is spawned. Coordinator boots correctly (F5 works — no tmux fallback), writes pid file, but exits after first tick because `NamedPipeClient::connect` fails (shim already dead).
- **Root cause**: Shim is spawned by quick-start's one-shot process. When quick-start exits, its child processes (worker + shim) also exit (Windows default). By the time operator manually starts `coordinator`, the shim is gone.
- **Fix options**:
  - **A**: Shim-as-service — install `windows-shim.exe` as a Windows service with lifetime independent of quick-start. Adds installer complexity.
  - **B**: Coord manages shim spawn — move `spawn_shim_and_handshake` from `lifecycle/launch.rs` into `coordinator::run_daemon`. Coord's own lifetime protects the shim.
  - **C**: Shim uses `DETACHED_PROCESS + CREATE_BREAKAWAY_FROM_JOB` on spawn so it truly survives its parent's exit (Windows-specific detach). Least invasive.
- **Recommendation**: Option C for the next batch — smallest change, no installer impact, doesn't require re-plumbing lifecycle.
