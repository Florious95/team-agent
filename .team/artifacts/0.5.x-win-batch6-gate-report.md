### CHECK 2: `quick-start --backend conpty` ⚠️ DESIGN GAP DISCOVERED
**Command**:
```
team-agent.exe quick-start --workspace <ws> --team team-a --backend conpty --yes --no-display --json
```

**Result**:
```json
{
  "error": "transport error: mux unavailable on ConPty: no_pipe_client (op=spawn, workspace_hash=f732217e7e92, team_key=team-a); no live shim wired — this is honest degradation, not silent success",
  "ok": false
}
```

**Path analysis** (progressed through each layer):
1. `TEAM.md` + `agents/worker-a.md` compile ✅ (spec compile succeeded)
2. Team `--team team-a` selection ✅
3. `--backend conpty` factory choice ✅ (`transport.kind=conpty` locked)
4. Worker spawn attempt → `TransportError::MuxUnavailable` — exactly the CR C-1 fail-closed path Batch 5 designed

**NOT hitting the tmux gate**:
- `lifecycle/display.rs::probe_display_capabilities` sets `platform_supported = !matches!(platform, "windows") && !in_wsl()`, so Windows lands on `AdaptiveBlockReason::NotImplementedThisPlatform`, NOT `LeaderNotInTmux`.
- There is no hard `must run inside tmux` gate in the launch/quick-start path.
- The "leader in tmux" concept is display-only, not a spawn precondition.

**Root cause of CHECK 2 fail**:
`quick-start` does not automatically spawn `windows-shim.exe`. The factory's Batch 5 `conpty_pipe_ready` gate expects either:
- `state.transport.shim.pipe_ready == true` (coordinator marks it after starting the shim), OR
- `TEAM_AGENT_FORCE_CONPTY_PIPE_CONNECT=1` env override.

Neither is present because the coordinator boot path never launches `windows-shim.exe`. When shim is running (verified separately with manual `Start-Process`), the factory still doesn't see the ready-hint in state and skips the connect, so ConPtyBackend degrades to MuxUnavailable at spawn.

**Minimal exemption fix proposal (for leader ruling)**:

Option A (simplest, coordinator-side spawn):
1. Add `crate::coordinator::conpty_shim` module: on `#[cfg(windows)]` + `resolved.kind == BackendKind::ConPty`, spawn `<install_dir>\windows-shim.exe --workspace-hash <hash> --team <team> --pipe-name <derived> --pipe-token <random>` as a child of the coordinator.
2. Write `state.transport.shim = { pid, pipe_name, pipe_ready: true, pipe_token }` after `Get-Process` confirms it's alive.
3. `quick-start` retries factory resolve — now `conpty_pipe_ready` gate opens, `NamedPipeClient::connect` succeeds, `ConPtyBackend` becomes wired.
4. `shutdown` reads `state.transport.shim.pid` and terminates the shim child cleanly (already covered by existing platform::process::terminate_pid path).

Scope: ~150 lines in `coordinator/`, no existing code path changes, no ambiguous semantics. Windows-only, cfg-gated. C-1 still enforced (no silent tmux fallback).

Option B (defer to Batch 7):
- Accept that Batch 6 delivers the "compile + status + honest MuxUnavailable" surfaces and treat the shim auto-spawn as Batch 7 scope. Users cannot yet run a full quick-start on Windows without manually starting the shim binary.

Option C (docs-only for Batch 6):
- Ship team-agent.exe with a `docs/windows-quickstart.md` explaining manual shim launch. Batch 7 automates it.

Recommendation: **Option A** — minimal code, closes the six-check loop, keeps CR C-1 fail-closed intact. Awaiting leader ruling.

## Findings not on the six-check list

### F1: Shim binary manual-spawn works
- `windows-shim.exe --workspace-hash <hash> --team <key> --pipe-name <\\.\pipe\...> --pipe-token <tok>` starts, writes stderr:
  ```
  windows-shim: workspace_hash=... team=... pipe=\\.\pipe\... pid=... version=0.1.0
  windows-shim: pipe_token=...
  windows-shim: waiting for client on \\.\pipe\...
  ```
- Confirmed via `Get-Process windows-shim` immediately post-start.
- The C-1 fail-closed at CHECK 2 IS the honest degradation Batch 5 designed for. Not a bug in Batch 5.

### F2: SSH escaping of `\\.\pipe\...` requires PowerShell single-quoted arguments
When shell-escaping `\\.\pipe\team-agent-conpty-...` through SSH → bash → PowerShell, double-quoting the arg mangles the leading `\\.` (drops one backslash). Fix: use PS single-quotes + a `$pipe` variable set inside PS. Documented for future Batch 6/7 harness authors.

### F3: `--backend conpty` fail-closed without `--team` is byte-preserving Batch 5 behavior
`quick-start --workspace <ws> --backend conpty` (no `--team`) → `spec compile failed: validation error` → correctly refuses without silent tmux fallback. C-1 held on Windows binary.
