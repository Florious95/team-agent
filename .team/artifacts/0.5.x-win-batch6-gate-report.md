## Additional findings

### F4: Phase 1b `windows-shim.exe` client-reuse race (PRODUCT BUG)
- **Symptom**: after first client Hello + disconnect, shim's next `create_named_pipe → ConnectNamedPipe` returns `ERROR_PIPE_CONNECTED (0x80070217)`. Subsequent clients fail to connect.
- **Root cause**: Phase 1b `windows_shim.rs::run()` outer loop close-open pattern lacks a cleanup barrier + reuses the same pipe name for the next instance.
- **Batch 6 workaround**: launch.rs bypasses factory re-connect entirely. The single Hello-connected `NamedPipeClient` is handed directly to `ConPtyBackend::with_pipe_client`. This means:
  - Worker spawn works from the initial launch client.
  - Subsequent CLI invocations (`send`, `peek`, `status --detail` after shim exit) hit `no_pipe_client` because they can't reconnect.
- **Not fixed in this batch**: fixing the shim's pipe reuse requires Phase 1c-scope changes to `windows_shim.rs::run()`. **Recommended for Batch 7 or dedicated shim-hardening batch**.

### F5: Coordinator daemon on Windows uses tmux fallback
- On explicit `team-agent coordinator --workspace ...` (no `--team`), the coordinator factory refuses ConPty (missing team_key) and logs "falling back to tmux workspace for daemon liveness".
- Impact: even with a live ConPTY worker session, the coordinator daemon isn't wired to the shim; MCP endpoints don't come up; `send` returns `coordinator_unavailable`.
- Fix scope: pass `--team` through to coordinator, OR persist team_key in state and read at coordinator boot. Batch 7 target.

### F6: state.transport.shim block gets clobbered by downstream state saves
- `finalize()` writes `state.transport.shim = {pid, pipe_name, pipe_ready: true}` correctly.
- Downstream state saves in quick-start don't merge my block through — final state.json has `transport.{kind, source}` only, no `shim`.
- **Workaround**: set env `TEAM_AGENT_FORCE_CONPTY_PIPE_CONNECT=1` in-process AND hand client directly. Both worked around the state loss but neither is ideal for CHECK 6 shutdown routing.
- **CHECK 6 impact**: shutdown returns `{action: "no_shim_recorded"}` — technically honest (state is what it is) but shim isn't reaped because pid was never persisted.
- Fix scope: modify state merge to preserve nested `transport.shim`, OR emit an event that shutdown scans. Batch 7 target.
